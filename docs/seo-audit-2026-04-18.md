# SEO Audit ‚Äî BookingTours (bookingtours.co.za)

**Date:** 2026-04-18
**Scope:** `/Users/gideonlangenhoven/Desktop/CapeKayak/capekayak` ‚Äî the Next.js app that serves both the BookingTours admin dashboard and the (minimal) public marketing surface.
**Method:** repo-level static audit only. No Lighthouse, Search Console, or live crawl yet ‚Äî those happen after Phase 1 is deployed.

---

## Executive summary

BookingTours is currently **~5% ready for SEO**. Google can crawl three public pages (`/case-study/cape-kayak`, `/compare/manual-vs-disconnected-tools`, `/operators`) but:

- `/` ‚Äî the site root ‚Äî is a gated admin dashboard, so Google's first impression of the domain is a loading spinner that resolves into a login screen.
- Every marketing page inherits an "Admin"-titled root metadata, no per-page metadata, no JSON-LD schema, no canonical URLs, and no Open Graph images.
- There is **no** `robots.ts`, `sitemap.ts`, `llms.txt`, or `opengraph-image`. Google has zero hints about what to crawl or ignore.
- The product is publicly referred to as "CapeKayak SaaS" on marketing pages, while the domain and brand is BookingTours. This is a visible brand inconsistency in the current public HTML.
- `public/landing-pages/` ships **87 MB** of tenant template assets from the same origin. They may be being crawled and counted against your domain's quality signals.

Nothing here is irrecoverable. All findings are in this repo; fixing them is ~3‚Äì4 days of work. The rest of the SEO playbook (content, keywords, backlinks) cannot meaningfully begin until the P0 items below are shipped.

## P0 ‚Äî blockers (must ship before any content work)

### P0-1 ¬∑ No public marketing home at `/`

**Evidence:** `app/page.tsx:1` is `"use client"` rendering the admin manifest/weather dashboard. `app/layout.tsx:43‚Äì45` wraps everything in `<AuthGate>` + `<AppShell>`. An unauthenticated visitor to bookingtours.co.za hits `AuthGate`, which on line 308 of `components/AuthGate.tsx` shows a "Checking admin session‚Ä¶" spinner, then renders a login form (line 327).

**Impact:** Google's top-ranking signal for a SaaS domain is usually the home page. Yours resolves to a login screen. There is literally nothing to index.

**Fix:** introduce a Next.js route group split.
- New: `app/(marketing)/page.tsx` ‚Äî public home.
- Move dashboard: everything currently in `app/` except `(marketing)`, `change-password`, `case-study`, `compare`, `operators`, `api`, `lib`, `favicon.ico`, `globals.css`, `loading.tsx`, `layout.tsx`, `page.tsx` goes into `app/(app)/`.
- New `app/(marketing)/layout.tsx` ‚Äî no AuthGate, no AppShell, dedicated marketing header/footer.
- New `app/(app)/layout.tsx` ‚Äî hosts the existing AuthGate + AppShell.
- Root `app/layout.tsx` becomes a slim shell: `<html><body><ThemeProvider>{children}</ThemeProvider></body></html>`.

### P0-2 ¬∑ Root metadata is titled "BookingTours Admin"

**Evidence:** `app/layout.tsx:11‚Äì15`:
```ts
export const metadata: Metadata = {
  title: "BookingTours Admin",
  description: "BookingTours Admin Dashboard ‚Äî Built for adventure operators",
  icons: { icon: "/favicon.ico" },
};
```

**Impact:** every crawled page ‚Äî including the current public case-study and compare pages ‚Äî inherits this title. SERP listings say "Admin". That is bad branding and bad CTR.

**Fix:**
- Root `app/layout.tsx` ‚Üí `title: { default: "BookingTours ‚Äî booking software for South African tour operators", template: "%s ¬∑ BookingTours" }`
- Add `metadataBase: new URL("https://bookingtours.co.za")`
- Add `alternates: { canonical: "/" }` per-page via `generateMetadata`
- Add `openGraph` + `twitter` defaults (OG image, site name, locale `en_ZA`)
- Add `robots: { index: true, follow: true }` to marketing routes and `robots: { index: false, follow: false }` to the `(app)` group layout so dashboard URLs don't get indexed
- Move the "Admin" wording into the `(app)` group layout metadata

### P0-3 ¬∑ No `robots.ts`, `sitemap.ts`, or `llms.txt`

**Evidence:** `find` returned nothing for `robots.ts`, `sitemap.ts`, `llms.txt`, `robots.txt`, `sitemap.xml`, `opengraph-image*`, or `proxy.ts`.

**Fix ‚Äî three files:**

1. **`app/robots.ts`** ‚Äî static Next.js metadata route. Allow all crawlers on marketing paths; disallow `/bookings`, `/settings`, `/inbox`, `/super-admin`, `/admin`, `/slots`, `/refunds`, `/invoices`, `/vouchers`, `/weather`, `/photos`, `/broadcasts`, `/pricing` (old dashboard path ‚Äî will be renamed), `/reports`, `/marketing` (dashboard route, NOT marketing content), `/new-booking`, `/change-password`, `/google-callback`, `/api/*`.
2. **`app/sitemap.ts`** ‚Äî programmatic, enumerates all `(marketing)` routes + any future MDX blog posts.
3. **`public/llms.txt`** ‚Äî per the llms.txt proposal, lists canonical URLs of key marketing pages to help LLM-based search surface the right pages. Small win, cheap to do.

### P0-4 ¬∑ No per-page metadata or schema

**Evidence:** `Grep "generateMetadata|export const metadata"` returned exactly one file ‚Äî `app/layout.tsx`. Zero marketing pages set their own metadata. Zero JSON-LD anywhere.

**Impact:** every marketing page has identical title + description in SERP. No rich results (FAQ, breadcrumb, software application, organization) will appear. This is the single largest "free" SEO win left on the table.

**Fix (part of Phase 2):**
- Every `(marketing)/*/page.tsx` exports `generateMetadata` with page-specific title, description, canonical, OG image.
- `(marketing)/layout.tsx` injects `Organization` + `WebSite` JSON-LD on every page.
- Vertical + feature pages inject `SoftwareApplication` + `FAQPage` JSON-LD.
- Case study injects `Article` + `BreadcrumbList` JSON-LD.
- Comparison pages inject `FAQPage` + `BreadcrumbList`.

### P0-5 ¬∑ Product brand inconsistency in public HTML

**Evidence:**
- `app/case-study/cape-kayak/page.tsx:9` ‚Äî H1 reads "How Cape Kayak runs booking‚Ä¶" (fine, this IS a case study)
- `app/compare/manual-vs-disconnected-tools/page.tsx:27` ‚Äî table header is "CapeKayak SaaS"
- `app/operators/page.tsx:21` ‚Äî pill badge says "CapeKayak SaaS"

**Impact:** Google indexes "CapeKayak SaaS" as a product name. Branded searches for "BookingTours" won't find these pages as strongly. Confusing for prospects.

**Fix:** global rename in all public-facing copy: `CapeKayak SaaS` ‚Üí `BookingTours`. Keep "Cape Kayak" (two words) only when referring to the tenant (the kayak business) in the case study. This is a 3-file edit.

## P1 ‚Äî foundations (ship immediately after P0)

### P1-1 ¬∑ AuthGate is in the root layout as a client component

**Evidence:** `components/AuthGate.tsx:1` `"use client"` + `app/layout.tsx:43` wraps all children. AuthGate imports supabase, sha256, admin-auth, localStorage code (~8 `localStorage` keys), and sends auth/login fetches. All of that JS loads on every public marketing page too.

**Impact:** Core Web Vitals. Marketing pages ship ~150 KB of unnecessary JavaScript (estimate ‚Äî actual TBD once I measure). LCP on mobile will suffer. TTI delayed by auth session checks that never apply to marketing visitors.

**Fix:** route-group split (P0-1) naturally solves this ‚Äî AuthGate moves into `(app)/layout.tsx`.

### P1-2 ¬∑ No image optimization anywhere in `app/`

**Evidence:** `Grep "next/image"` ‚Üí zero matches in `app/`. `Grep "<img"` ‚Üí 4 matches (`super-admin`, `photos`, `settings`, `new-booking` ‚Äî all dashboard routes). `public/pictures/` has 1.8 MB of raw JPEGs named like `media__1772174067696.jpg`.

**Impact:** today this is low-risk because marketing pages don't use images. But as soon as Phase 3 ships hero images on vertical pages, any `<img>` tag = CWV regression.

**Fix:** enforce `next/image` via an ESLint rule (`@next/next/no-img-element`) in `eslint.config.mjs`. Pre-size every image. All hero images and feature screenshots go in `public/marketing/` with explicit width + height attributes.

### P1-3 ¬∑ 87 MB `public/landing-pages/` deploys to the SaaS origin

**Evidence:** `du -sh public/landing-pages` = 87M. Inside are 9 HTML templates + fonts + images + videos intended to be the template library for tenants to clone.

**Impact:** these static HTML files deploy to the main bookingtours.co.za origin and are crawlable by default. They contain template copy (adventure/luxury/safari/etc.) that Google can index as *your* content ‚Äî creating keyword cannibalization (Google can't tell which of your 9 "adventure" pages is the canonical one) and thin-content flags (templates full of Lorem Ipsum or placeholder text).

**Fix ‚Äî pick one:**
- (a) Move to a separate subdomain (`templates.bookingtours.co.za`) with its own `robots.txt` disallowing all crawlers. Cleanest.
- (b) Keep at current URL but add `X-Robots-Tag: noindex, nofollow` for `/landing-pages/*` via `next.config.ts` headers.
- (c) Move off the origin entirely (S3/R2 bucket).

Before picking, confirm: how are tenants currently using these? Is there a live URL that points here? I want to see the usage before suggesting the change.

### P1-5 ¬∑ `bookings/[id]/page 2.tsx` is a real duplicate source file

**Evidence:** the earlier "use client" grep flagged `app/bookings/[id]/page 2.tsx` alongside the real `app/bookings/[id]/page.tsx`. Git status shows it as untracked. This is a dashboard route duplicate, not a marketing one ‚Äî but Next.js will refuse to build or will produce unpredictable route resolution.

**Fix:** diff it against `page.tsx`; keep the newer version; delete the other. Verify `npm run build` succeeds.

### P1-6 ¬∑ Viewport, theme color, and language metadata missing

**Evidence:** `app/layout.tsx` sets `lang="en"` but the target market is South Africa ‚Äî should be `lang="en-ZA"`. No `viewport`, no `themeColor`.

**Fix:**
- `lang="en-ZA"` on `<html>`.
- Root metadata: `viewport: { width: "device-width", initialScale: 1 }`, `themeColor: "#0f172a"` (or brand colour).
- `manifest: "/site.webmanifest"` + create a minimal PWA manifest.

### P1-7 ¬∑ No middleware / proxy to separate marketing from app

**Evidence:** no `middleware.ts` or `proxy.ts` in repo root. Next.js 16 recommends `proxy.ts` (the renamed middleware primitive). See `https://nextjs.org/docs/app/api-reference/file-conventions/middleware` and the Vercel chunk at session start.

**Opportunity, not a blocker:** once you add a marketing subdomain (recommended: run the marketing site at `bookingtours.co.za` and the app at `app.bookingtours.co.za`), you'll want host-based routing in `proxy.ts`. But this is P2 ‚Äî the route-group split (P0-1) is enough to unblock indexing.

## P2 ‚Äî polish (ship after content is live)

### P2-1 ¬∑ No `opengraph-image.tsx`

Next.js can auto-generate per-route OG images. `app/opengraph-image.tsx` + one per marketing route = branded social shares for every page without any design work.

### P2-2 ¬∑ No Core Web Vitals instrumentation

Install `@vercel/speed-insights` and `@vercel/analytics`. Both are free on Vercel Hobby/Pro. Gives per-route RUM data in the Vercel dashboard. Without this you cannot tell which pages are slow.

### P2-3 ¬∑ No sitemap revalidation

The `sitemap.ts` should set `export const revalidate = 3600` so it refreshes hourly as blog content publishes. Or use on-demand revalidation via Vercel Queues once blog publishing is automated.

### P2-4 ¬∑ `next.config.ts` doesn't configure image domains or cache

**Evidence:** `next.config.ts` only sets security headers. No `images` config. If you ever `<Image src="https://supabase-bucket-url/...">`, it'll fail.

**Fix:** add `images: { remotePatterns: [{ protocol: "https", hostname: "<supabase-storage-host>" }] }`. Add marketing-specific cache headers:
```ts
{ source: "/:path(\\(marketing\\).*)", headers: [{ key: "Cache-Control", value: "public, s-maxage=86400, stale-while-revalidate=604800" }] }
```

### P2-5 ¬∑ `/pricing` path collision

**Evidence:** `app/pricing/page.tsx` is the dashboard "Peak Pricing" tool (`"use client"`, connects to Supabase). That claims the ideal SEO slug for the public pricing page.

**Fix:** rename dashboard tool to `/admin/peak-pricing` (or similar under the `(app)` group). Reserve `/pricing` for the public pricing page.

### P2-6 ¬∑ `/operators` is a poor public URL

**Evidence:** `app/operators/page.tsx` dual-purposes as marketing (unauthenticated view) and operator switcher (authenticated). The URL slug `/operators` reads as a directory of operators to Google, not a pricing/signup page.

**Fix:** after `(marketing)/page.tsx` + `(marketing)/pricing/page.tsx` exist, redirect `/operators` (for unauthenticated users) to `/pricing` and keep `/operators` strictly as the authenticated switcher.

### P2-7 ¬∑ No analytics or Search Console verification

Add `google-site-verification` meta tag, GA4 snippet, and Plausible/PostHog if desired. All of this is trivial once the route-group split is in place.
